# ML Distinguisher Experiment

This directory contains a PyTorch classifier that trains on raw N-byte
outputs from the fast cipher implementations in this repository.

The paired `*Ct` variants are intentionally excluded from training because they
should produce exactly the same bits as the fast versions for the same key and
input. Including them as separate classes would only test for harness bugs or
label leakage.

## Dataset

The dataset is generated by the Rust binary:

```sh
cargo run --release --bin gen_ml_dataset -- --output ml/data
```

You can increase the sample width for stronger aggregation tests:

```sh
cargo run --release --bin gen_ml_dataset -- --output ml/data --sample-len 256
```

Each sample is exactly `sample_len` bytes. The generator writes:

- `train_samples.bin`, `val_samples.bin`, `test_samples.bin`
- `train_labels.bin`, `val_labels.bin`, `test_labels.bin`
- `manifest.json`

Default classes:

- `aes128`
- `aes256`
- `des`
- `tdes3`
- `simon128_128`
- `speck128_128`
- `magma`
- `grasshopper`
- `sm4`
- `zuc128`
- `random`

## Training

Use the local PyTorch virtualenv:

```sh
ml/.venv-torch/bin/python ml/train_distinguisher.py --generate
```

Two useful knobs for longer experiments:

```sh
ml/.venv-torch/bin/python ml/train_distinguisher.py \
  --generate \
  --sample-len 256 \
  --model-size large
```

Try the patch-based Transformer instead of the CNN:

```sh
ml/.venv-torch/bin/python ml/train_distinguisher.py \
  --generate \
  --sample-len 256 \
  --architecture transformer \
  --model-size large \
  --patch-len 16
```

To recreate that environment later:

```sh
python3 -m venv ml/.venv-torch
ml/.venv-torch/bin/python -m pip install -r ml/requirements.txt
```

On Apple silicon, the trainer will use the MPS backend automatically when
PyTorch can see the GPU. In this Codex sandbox, GPU visibility is blocked, so
GPU-backed training from here requires running the process outside the sandbox.

Artifacts are written to `ml/out/`:

- `cipher_distinguisher.pt`
- `cipher_distinguisher_state_dict.pt`
- `labels.json`
- `metrics.json`
- `history.csv`

The trainer supports:

- `--architecture cnn|transformer|byte_transformer`
- `--model-size base|large|xlarge`

`transformer` is a patch Transformer (good for wide samples). `byte_transformer`
attends over every byte token directly, which is more expensive but gives a
different inductive bias for subtle sequence-level effects.

## Adaptive Overnight Runs

Use the staged runner to adapt sample width and model size over a long run:

```sh
ml/.venv-torch/bin/python ml/run_adaptive_experiment.py
```

It runs a default sequence of stages, reads each `metrics.json`, and:

- escalates when held-out accuracy stays near chance
- repeats the same stage with fresh seeds if a run rises materially above chance
- stops early when a candidate signal survives confirmation

## Dataset Checks

Audit a generated dataset before trusting any surprising model result:

```sh
python3 ml/check_dataset.py --data-dir ml/data
```

The checker reports:

- row counts and label balance
- duplicate rows within each split
- overlap between train/validation/test
- simple byte-distribution sanity stats
